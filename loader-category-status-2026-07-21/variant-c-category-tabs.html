Loader — in-progress · punch-list · close-out gate (Variant C)

Three-part loader UI. Category is metadata + a filter dimension, never a nesting level — hierarchy depth stays at three.

Product goal — eliminate secondary delivery runs. Today loaders don't verify against paper, so missed tags send a truck back. This design's job is to create friction at the departure moment.

Hierarchy: Delivery run = bin X8 → Order = bin.suffix (X8.1, X8.2, X8.3) → status-grouped tags. Category rides along as the "where do I walk" dimension.

How the three parts work together. The in-progress view is where the loader lives; the punch list is the same data pivoted into a walking route; the close-out gate is the only new hard stop.

Per-tag photo — 📷 tap marks rows where tapping opens the camera, asks you to confirm you loaded it, and files the photo against that tag. It is the hand-load path for a tag whose barcode won't scan, and it always ends with the tag marked Loaded. Rows that aren't finished yet (At machine / Not printed) don't offer it.

Who else is on this run — the header shows other active loaders and how fresh the data is; loaded rows carry a quiet "Loaded by Jose · 4s ago". A row someone else just completed pulses briefly. If two people scan the same tag, the second gets told who got there first rather than a silent no-op.

Switching runs — the ⇄ button beside the run number opens the picker (frame 6).

Mobile — on a phone this page drops the simulated phone frame and fills the screen; the tabs at the top jump between frames. On desktop the side-by-side frames are unchanged.
